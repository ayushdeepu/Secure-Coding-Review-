Methodology

The application was reviewed using a combination of static analysis and manual line-by-line inspection of all request handlers, database queries, and file I/O.

Manual Inspection

Each route was traced from user input to its final sink (SQL query, template render, or filesystem call) to identify untrusted data reaching a dangerous operation.

Static Analysis — Bandit

Bandit was run against the codebase to flag known insecure patterns automatically.

Detailed Findings & Remediation

Each finding includes the vulnerable code, the risk it introduces, and the corrected implementation.

Critical

F1 — SQL Injection

/login · CWE-89

User-supplied username and password are concatenated directly into a SQL string, letting an attacker bypass authentication using a payload such as ' OR '1'='1.

Vulnerable
query = (
  "SELECT * FROM users WHERE "
  "username = '" + username +
  "' AND password = '" + password + "'"
)
cur.execute(query)
Fixed
query = (
  "SELECT password FROM users "
  "WHERE username = ?"
)
cur.execute(query, (username,))
Critical

F2 — Plaintext Password Storage

/register · CWE-256

Passwords are stored as-is in the database. A single database leak would expose every user's real password, which is often reused across other services.

Vulnerable
cur.execute(
  "INSERT INTO users (username, password) "
  "VALUES (?, ?)",
  (username, password)   # plaintext
)
Fixed
from werkzeug.security import \
  generate_password_hash

hashed = generate_password_hash(password)
cur.execute(
  "INSERT INTO users VALUES (?, ?)",
  (username, hashed)
)
High

F3 — Reflected XSS

/profile · CWE-79

The name query parameter is concatenated into raw HTML and rendered without escaping, allowing an attacker to inject <script> tags via a crafted link.

Vulnerable
name = request.args.get("name")
template = (
  "<h1>Welcome, " + name + "</h1>"
)
return render_template_string(template)
Fixed
name = request.args.get("name", "")
return render_template(
  "profile.html", name=name
)
# profile.html: <h1>Welcome, {{ name }}</h1>
# Jinja2 auto-escapes {{ name }}
High

F4 — Path Traversal

/download · CWE-22

The file parameter is joined directly onto the uploads path. A value like ../../etc/passwd escapes the intended directory and reads arbitrary files on the server.

Vulnerable
filename = request.args.get("file")
path = os.path.join("uploads", filename)
with open(path) as f:
    return f.read()
Fixed
from werkzeug.utils import secure_filename

UP = os.path.abspath("uploads")
fn = secure_filename(request.args.get("file",""))
path = os.path.join(UP, fn)
if not os.path.abspath(path).startswith(UP):
    return "Invalid file", 400
High

F5 — Debug Mode Enabled

app.run() · CWE-489

Running Flask with debug=True exposes the Werkzeug interactive debugger. If reachable externally, this can allow remote code execution and leaks stack traces with source code and environment variables.

Vulnerable
app.run(debug=True, host="0.0.0.0")
Fixed
app.run(debug=False)
# control via env var, never hardcode True
# DEBUG = os.environ.get("FLASK_DEBUG", "0") == "1"
Medium

F6 — Missing Input Validation

all routes · CWE-20

None of the routes validate length, type, or format of incoming data, which increases the attack surface for injection, buffer issues, and malformed-data bugs.

Vulnerable
username = request.form['username']
password = request.form['password']
# no checks at all
Fixed
username = request.form.get('username','')
password = request.form.get('password','')
if not username or not password \
   or len(username) > 50:
    return "Invalid input", 400

Secure Coding Best Practices

General guidelines to apply across the codebase, beyond the specific fixes above.

01

Parameterized queries only — never build SQL via string concatenation or f-strings.

02

Hash passwords with bcrypt, Argon2, or PBKDF2 — never store them in plaintext.

03

Use template auto-escaping (Jinja2 render_template) instead of raw string HTML.

04

Sanitize file paths with secure_filename() and verify the resolved path stays inside the allowed directory.

05

Disable debug mode in production; drive it from environment configuration.

06

Validate all input — length, type, and format — before using it.

07

Set secure cookie flags: HttpOnly, Secure, SameSite.

08

Run SAST in CI — Bandit or Semgrep on every pull request.

09

Rate-limit auth endpoints to mitigate brute-force attacks.

10

Keep dependencies current — scan with pip-audit or safety.
